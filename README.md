## 🔎 إصلاح توقّف الفهرسة على Google بعد الانتقال إلى Cloudflare (Phase B)

بعد الانتقال إلى Cloudflare توقّف زحف Google فجأة (كان ≈4000/يوم) ولم تعد الصفحات
تُفهرَس حتى عند طلب الفهرسة يدوياً. التشخيص والإصلاحات:

**السبب الجذري — ملف `_routes.json` المُولَّد تلقائياً:** محوّل `@astrojs/cloudflare`
يولّد `dist/_routes.json` لتوجيه طلبات الـ Functions. لأن عدد الصفحات الثابتة
(≈5,800) أكبر بكثير من حدّ Cloudflare (100 قاعدة)، كان المحوّل يسقط في فرع الاقتطاع
فيكتب `include: ["/*"]` مع قائمة `exclude` مبتورة. النتيجة: **كل طلب** (كل صفحة ثابتة
وكل أصل) يستدعي الـ Worker، ما يستنزف حصّة الـ Functions اليومية بسرعة (زحف Google
وحده كان يكفي)، فتفشل الصفحات في الخدمة ويتوقف Google عن الفهرسة. صفحات الحلقات SSR
تتنافس على نفس الـ Worker المُجهَد فتتراكم حالة «Crawled - currently not indexed».

- **الحل:** سكربت ما بعد البناء `scripts/fix-routes-json.mjs` يعيد كتابة
  `dist/_routes.json` بقواعد دقيقة: `include` يقتصر على مسارات الـ SSR الحقيقية فقط
  (`/d/* /n/* /w/* /gw/* /g/* /x/c/* /episodes-sitemap*` + `/_image` + `/_server-islands/*`)
  و`exclude` للأصول/الصفحات الثابتة. الإجمالي **36/100 قاعدة** (لا اقتطاع). بذلك
  تُخدم الرئيسية والأفلام والقوائم والأصول كصفحات ثابتة (مجانية وسريعة وHTML صحيح)
  ولا يُستدعى الـ Worker إلا للمسارات الديناميكية فعلاً. مربوط في `npm run build`.

**إصلاح ثانٍ — Service Worker إعلاني ضار (`public/sw.js`):** كان يحمّل سكربتاً خارجياً
من `5gvci.com` ويعترض كل طلبات الموقع (سلوك تُعامله محرّكات البحث كـ cloaking وقد يكبح
الفهرسة). تم استبداله بـ Service Worker **يلغي تسجيل نفسه** ويمسح أي كاش قديم تلقائياً،
ولا يعترض أي طلب شبكة.

**إصلاح ثالث — المحتوى المكرّر مقابل الموقع الأقدم:** الموقع الأقدم المفهرس يستخدم
نفس نصوص القصص المسحوبة. أُضيفت دالة `uniquifyPlot()` في `src/lib/detailContent.ts`
تُعيد صياغة فقرة القصة المرئية + الـ meta description + وصف JSON-LD بمقدّمة وخاتمة
عربية متغيّرة لكل عمل (مُولّدة من سلَج العمل، ثابتة وحتمية) — تحافظ على المعنى لكنها
تجعل النص فريداً فلا يتطابق حرفياً مع الموقع الأقدم. مطبّقة على الصفحات الثابتة
(`/f /d /n`) وصفحات `/w` الـ SSR. عدد الأعمال ذات القصة الحقيقية المتأثرة: **8,280**.

**حذف 27 عملاً (delet..txt):** طُبّق عبر `scripts/apply-deletions.mjs` (27/27 تطابق)،
فانخفض الكتالوج من 18,559 → **18,532**، وأُعيد بناء الشاردات.

## 🚀 إصلاح فشل النشر على Cloudflare Pages (حدّ 20,000 ملف)

فشل النشر على Cloudflare Pages بخطأين (من ملف log النشر):

- **السبب:** كل عمل جديد (≈12,829) كان يُنتج ملف شاردة منفصل، كما أن البوابة القديمة `/g` كانت تُبنى صفحة ثابتة لكل عمل (≈5,814) — فتجاوز إجمالي `dist` حدّ الـ 20,000 ملف.
- **الحل (طبقتان):**
  1. **تجميع الأعمال الجديدة في 256 ملف bucket** (`public/_data/new/<n>.json`) عبر دالة هاش FNV-1a ثابتة (`NEW_BUCKETS=256`) متطابقة بين سكربت البناء و `src/lib/newWorks.ts`. النتيجة: 12,829 ملف → 256.
  2. **تحويل البوابة القديمة `/g` إلى SSR** (`prerender = false`) مثل `/gw`، مع 256 ملف bucket (`public/_data/oldgw/<n>.json`) تُجلب وقت الطلب عبر `src/lib/oldGateway.ts`. النتيجة: 5,814 صفحة ثابتة → 0.
- **النتيجة النهائية:** ميزانية `dist` انخفضت من ≈21,100 → **≈15,557 ملف** (هامش آمن ≈4,400 تحت الحد). تم التحقق من تطابق الهاش (823/823 + 660/660) واختبار البوابات SSR على wrangler (200/404 كما هو متوقع).

**المشكلة 2 (تحذير شكلي غير قاتل):** `wrangler.json file ... does not appear to be valid`.
- **السبب:** يوجد ملف `wrangler.json` (وليس `.jsonc`) **غير صالح في مستودع GitHub الخاص بك** — مشروعنا المحلي يستخدم `wrangler.jsonc` الصحيح فقط.
- **الإجراء المطلوب منك:** احذف ملف `wrangler.json` من مستودع GitHub (`mohmed-git/editon`) وأبقِ `wrangler.jsonc` فقط. هذا التحذير لا يوقف النشر لكن حذفه ينظّف السجل.

## 🆕 ترقية معمارية: Astro Static → Astro Hybrid (SSR لصفحات الحلقات)

تم تحويل المشروع من **Static كامل** إلى **Hybrid** على Cloudflare Pages:

- **الافتراضي: Static** — كل الصفحات تبقى مُولّدة مسبقاً (الرئيسية، القوائم `/x/*`، صفحات الأفلام `/f/*`، صفحات المسلسلات/الأنمي `/d|n/[slug]`، صفحات المواسم `/d|n/[slug]/c/[season]`، البوابة `/g/*`). السبب: عددها محدود (~9 آلاف) ونادرة التغيّر، فالـ static يعطي أقصى سرعة وأفضل SEO.
- **استثناء: صفحات الحلقات SSR** — `/d/[slug]/c/[season]/e/[episode]` و `/n/...` أصبحت **on-demand SSR** (`export const prerender = false`). السبب: عددها **48,174 حلقة**، ولو بُنيت static لتجاوزت حد **20 ألف ملف** في Cloudflare Pages (وهو سبب حذفها سابقاً في `fix-episode-pages.patch`). الآن تُولَّد عند الطلب وتُكاش على الـ edge، فلا ملفات إضافية في `dist`.

### كيف تُحمّل بيانات الحلقات في SSR دون تضخيم الـ Worker
- `all.json` (86MB) **لا يُحزَّم** داخل Worker الـ SSR (كان يسبب OOM ويتجاوز حد حجم الـ Worker).
- سكربت `scripts/build-episode-shards.mjs` (streaming، بلا اعتماديات) يقسّم العناوين الحلقية إلى **2,422 shard** صغير تحت `public/_data/episodes/<base64url(slug)>.json` كأصول static، إضافةً إلى:
  - `episode-manifest.json` (slug → اسم الملف) — صغير ويُحزَّم.
  - `episode-routes.json` (~270KB) — slug/category/أرقام الحلقات، لتوليد الـ sitemap.
- عند الطلب: `src/lib/episodeData.ts` يجلب **shard واحد فقط** عبر `fetch('/_data/episodes/...')` من نفس النشر.

#### 🛠️ إصلاح فشل الرفع على Cloudflare (Failed to publish assets)
- **المشكلة**: كان `src/lib/similarLite.ts` يستورد `index.json` (~5MB) **بشكل ثابت (static import)**، فتم تحزيمه داخل Worker الـ SSR وأنتج chunk بحجم **5,609 KiB** وإجمالي Worker **6,670 KiB** — وهو ما تجاوز حدود Cloudflare وأدى إلى `Failed: an internal error occurred` / `Failed to publish assets` رغم نجاح البناء.
- **الحل**: نفس نمط `episodeData.ts` — سكربت `build-episode-shards.mjs` صار يولّد فهرس "العناوين المشابهة" المُبسّط مقسّماً **حسب التصنيف** كأصول static: `public/_data/similar/{movie|series|anime}.json` (بحقول البطاقة فقط). و `similarLite.ts` يجلب ملف تصنيف العنوان الحالي فقط عبر `fetch()` وقت الطلب بدل تحزيم الفهرس كاملاً.
- **النتيجة**: chunk الـ `similarLite` هبط من **5,609 KiB → ~20 KiB**، وإجمالي `_worker.js` من **6,670 KiB → ~868 KiB** (تقليص ~7.7×) — أصبح ضمن حدود Cloudflare ويُرفَع بنجاح. حجم الـ Worker الآن **مستقل عن حجم البيانات** (كل البيانات أصول static تُجلب وقت الطلب).

#### 🛠️ إصلاح ثانٍ لفشل الرفع: `all.json` (86MB) داخل `_worker.js`
- **العَرَض**: البناء `npm run build` ينجح، لكن **النشر يفشل خلال ثوانٍ** على Cloudflare بالخطة المجانية (الخطأ النموذجي: `Your Worker exceeded the size limit of 3 MiB` — code `10027`، أو `Failed to publish assets`).
- **السبب الجذري**: ملف `src/lib/titles.ts` كان يستخدم **static import**: `import all from '../data/generated/all.json'`. كل الصفحات الـ static (`index`, `search`, `/x/*`, `/f/[slug]`, `/d|n/[slug]`, `/g/[token]`...) تستورد هذا الملف. رغم أنها صفحات **prerendered** (تُبنى وقت البناء فقط)، فإن Vite/Rollup يُحزِّم `all.json` في chunk مشترك (`titles_*.mjs` بحجم **54.8 MB**) يضعه الـ Cloudflare adapter داخل `dist/_worker.js`. النتيجة: حجم `_worker.js` = **55 MB** ≈ 18 ضعف حد الخطة المجانية → فشل النشر فوراً. (الـ tree-shaking لا يعرف أن البيانات مطلوبة وقت البناء فقط، فلا يحذفها من الـ Worker.)
- **الحل**: استبدال الـ static import بقراءة الملف من القرص عبر `node:fs` وقت البناء:
  ```ts
  import { readFileSync } from 'node:fs';
  import { resolve } from 'node:path';
  const ALL_JSON_PATH = resolve(process.cwd(), 'src/data/generated/all.json');
  const RAW_TITLES = JSON.parse(readFileSync(ALL_JSON_PATH, 'utf-8'));
  ```
  بما أن **كل** الصفحات المستوردة لـ `titles.ts` هي `prerender = true`، فقراءة `fs` تُنفَّذ في Node أثناء البناء فقط (لا على الـ edge)، وبما أنه لا يوجد static import فإن الـ bundler **لا يُحزِّم `all.json` في أي chunk** إطلاقاً. *(ملاحظة: المسار يُحلّ عبر `process.cwd()` وليس `import.meta.url`، لأن الـ adapter ينسخ وحدات الصفحات إلى `dist/_worker.js/` قبل تمريرة الـ prerender.)*
- **النتيجة**: `_worker.js` هبط من **55 MB → 1.5 MB** (وبعد gzip ≈ **0.29 MB**)، واختفى chunk الـ 54.8MB تماماً — أصبح النشر ينجح. حجم الـ Worker الآن مستقل عن حجم `all.json` مهما كبرت المكتبة.

> ⚠️ **تنبيه عدد الملفات**: البناء الكامل ينتج حالياً **~17,540 ملف static** (حد Cloudflare Pages = **20,000**). أنت قريب من الحد — مع نمو المكتبة قد تتجاوزه. عند الاقتراب، انقل المزيد من الصفحات منخفضة الحركة إلى SSR (مثل صفحات الأفلام `/f/[slug]`) بنفس نمط الحلقات، أو استضف الأصول الكثيرة على R2.

### SEO وكاش صفحة الحلقة
- `title` + `description` + `canonical` + Open Graph (`og:type=video.episode`) + Twitter + **JSON-LD `TVEpisode`** — كلها مُولّدة من الخادم (HTML كامل قابل للفهرسة).
- `Cache-Control: public, max-age=0, s-maxage=3600, stale-while-revalidate=86400` (كاش edge لساعة + تقديم نسخة قديمة لمدة يوم أثناء التحديث).
- روابط غير الموجود (slug/حلقة خاطئة) ترجع **404 حقيقي** (لا soft-404).

### Sitemap للحلقات
- `/episodes-sitemap.xml` = فهرس sitemap (SSR) يشير إلى أجزاء `/episodes-sitemap/{n}.xml` (كل جزء ≤ 10,000 رابط).
- مربوط بالفهرس الرئيسي عبر `customPages` في `astro.config.mjs`، فتبقى الحلقات مشمولة في الـ sitemap رغم كونها SSR.

### الروابط الداخلية وصفحة الموسم
- `getNavigableEpisodeRouteForTitle` رجع ليشير إلى صفحة الحلقة المستقلة `/.../c/{season}/e/{episode}` بدل `?e=` السابق.
- أُزيل من `SeasonPage.astro` حقن تفاصيل الحلقة عبر JavaScript (`?e=`)؛ وأُلغيت قواعد `_redirects`/`robots` التي كانت تحوّل/تمنع صفحات الحلقات.

## معمارية البيانات
- **مصدر البيانات**: ملفات JSON لكل عمل في `src/data/titles/*.json` → يجمعها `scripts/build-data.mjs` إلى `src/data/generated/all.json`.
- **نموذج البيانات**: `Title` (slug, clean_title, category, poster, seasons[], episodes[], servers[], genre, year, quality, language ...).
- **ترميز الروابط**: مركزي في `src/lib/routes.ts` (detail/listing/season/episode + encode/decodeGateToken).
- **تحميل العناوين**: `src/lib/titles.ts` يعيد كتابة حقل `url` فقط إلى الشكل المشفّر (دون لمس الصور).
- **محتوى السيو/الوصف**: القصة الفعلية أولاً، ثم مولّدات نصّية حتمية عربية في `src/lib/detailContent.ts` للفقرات السياقية عند نقص البيانات، مع وصف احتياطي في `scripts/build-data.mjs` لتجنب المحتوى الرقيق.
- **الإثراء والترتيب**: كاش داخلي في `scripts/.cache/` يدمجه `build-data.mjs` ويولّد `sort_rating` (Bayesian) و`sort_recent` و`is_special` و`real_plot` دون إظهار أي نسب مصدر داخل الصفحات العامة.
  - سكربتات الإثراء تحتاج مفاتيح API في `.dev.vars` عند استخدامها محلياً.
- **حقول الترتيب**: `Title.{tmdb_vote,tmdb_votes,release_date,sort_rating,sort_recent,real_plot,is_special}` + نفسها في `TitleIndexEntry` لاستخدامها في الفرز من جهة العميل.

## دليل الاستخدام السريع
1. تصفّح الأقسام من القائمة العلوية (أفلام/مسلسلات/أنمي) — يتغيّر الثيم تلقائياً حسب القسم.
2. افتح صفحة عمل لرؤية القصة والمعلومات والأعمال المشابهة، ثم اضغط «شاهد الآن».
3. في بوابة المشاهدة بدّل بين السيرفرات والحلقات، أو استخدم «السابق/التالي».
4. استخدم `/search` للبحث الفوري الضبابي (fuse.js).

### ملاحظة المعاينة (هام)
بيئة المعاينة محدودة الذاكرة (~1GB)، وبناء 5800 عمل يولّد عشرات آلاف الصفحات ويتجاوز الذاكرة. لذلك:
- `src/data/generated/all.json` محفوظ حالياً بالكتالوج الكامل بعد تشغيل `npm run data` فقط، دون بناء كل الصفحات.
- يمكن اختبار البناء محلياً بمجموعة صغيرة عبر `PREVIEW_LIMIT=60 npm run build` أو قيمة قريبة، ثم إعادة `npm run data` لاسترجاع `all.json` الكامل.
- **النسخة الكاملة الاحتياطية (5814 عملاً، 93MB تقريباً) محفوظة في** `src/data/generated/all.full.json`.
- **للنشر الإنتاجي الكامل**: ابنِ على بيئة CI ذات ذاكرة أكبر بدلاً من sandbox محدود الذاكرة.
